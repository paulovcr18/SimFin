# Phase 03 Plan 01: Portfolio Performance

## Objective

Eliminar o gargalo O(days × tickers) em `build_evolution()` cacheando o resultado com `@st.cache_data` e corrigir a lógica de resolução de conflito de sync em `js/db.js` que pode reverter deleções do usuário. Também capturar erros de `maybeSingle()` no `_dbPullCarteira` e `_dbPullConfig`.

## Context

- `streamlit-app/pages/2_Carteira.py` — página da Carteira B3
- `streamlit-app/core/portfolio.py` — `build_evolution()` O(days × tickers)
- `js/db.js` — `_dbPullCarteira` e `_dbPullConfig` com lógica de sync problemática

## Tasks

### Task 1: Cache build_evolution() em 2_Carteira.py

**type:** auto

Cache o resultado de `build_evolution()` em `2_Carteira.py` usando `@st.cache_data` com TTL de 300s, keyed pelo hash SHA256 das transações serializadas. Isso evita recomputar O(days × tickers) a cada page load quando os dados não mudaram.

**Done criteria:**
- `build_evolution()` é chamado dentro de uma função decorada com `@st.cache_data`
- A cache key inclui o hash das transações (não apenas o uid)
- TTL de 300 segundos (5 minutos)
- Cache é invalidada quando `carteira_upsert` é chamado (st.cache_data.clear())

### Task 2: Fix sync conflict em _dbPullCarteira em js/db.js

**type:** auto

A lógica atual em `_dbPullCarteira` usa "quem tem mais entradas ganha" — isso pode reverter deleções do usuário. Corrigir para que o remote **sempre vença** durante o pull (o pull só ocorre no login, e o remote é a fonte de verdade). Remover a lógica de comparação de `length`.

Também adicionar tratamento de erro para `maybeSingle()` em `_dbPullCarteira` e `_dbPullConfig` — atualmente os erros são silenciosamente ignorados.

**Done criteria:**
- `_dbPullCarteira`: remote substitui local incondicionalmente (sem comparação de length)
- `_dbPullConfig`: remote substitui local incondicionalmente
- Erros de `maybeSingle()` são capturados e logados com `console.error`
- Erros de `carteira_posicoes` query são capturados e logados
